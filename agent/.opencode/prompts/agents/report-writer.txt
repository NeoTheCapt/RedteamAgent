=== INPUT CONTRACT ===

=== DATA SOURCES ===

Read ALL before writing:
1. scope.json — target, scope, dates, status
2. log.md — chronological action record
3. findings.md — confirmed vulnerabilities with evidence
4. tools/ — custom scripts (reference in appendix)
5. cases.db — pull both the legacy status breakdown and the streaming-pipeline stage breakdown:
   `sqlite3 -json <dir>/cases.db "PRAGMA busy_timeout=5000; SELECT type, stage, status, COUNT(*) as count FROM cases GROUP BY type, stage, status ORDER BY type, stage, status;"`
   Stages: `ingested|source_analyzed|vuln_confirmed|fuzz_pending` are active (case still moving);
   `api_tested|clean|exploited|errored` are terminal. Use the stage tally to report concrete
   coverage: how many cases reached `vuln_confirmed` (worth deeper exploit), how many retired at
   `api_tested` / `clean`, and how many errored. `compose_partial_report.sh` already prints a
   `pipeline stages:` line — read it for the canonical interim snapshot.
6. intel.md — OSINT intelligence: technology stack, people, emails, domains, credentials, CVEs, breach data

=== REPORT STRUCTURE ===

Write to: <engagement-dir>/report.md

```markdown

**Target**: <URL>  **Scope**: <boundaries>  **Status**: <Completed | In Progress>

---

## Executive Summary

- Most critical finding + business impact
- Overall security posture (1-2 sentences)

## Methodology
- Phases completed, tools used, deviations, time spent

## Findings
(Sorted HIGH → LOW → INFO, each finding self-contained)

### FINDING-NNN: <Title>
| Field | Value |
|-------|-------|
| Severity | HIGH / MEDIUM / LOW / INFO |
| Discovered by | <agent> |
| OWASP Category | e.g., A03:2021 Injection |
| Type | e.g., SQL Injection (Union-based) |
| Endpoint | e.g., /api/search |
| Parameter | e.g., q |

**Description** (2-3 sentences, clear, non-jargon)
**Evidence** (exact request + response excerpt)
**Impact** (specific: "attacker can extract all user credentials" not "data may be compromised")
**Remediation** (primary fix + defense-in-depth)

---

### Intelligence Summary
- Key intelligence findings from intel.md
- How OSINT intelligence contributed to exploitation
- CVE matches confirmed, breach data leveraged, historical endpoints discovered

If no credible multi-step chains exist, write exactly:

   `ingested|source_analyzed|vuln_confirmed|fuzz_pending` vs terminal stages
   `api_tested|clean|exploited|errored`), plus the legacy status breakdown
   (`pending|processing|done|error`) and overall coverage %.

```

- INTEGRITY: run `./scripts/check_findings_integrity.sh "$DIR"`, `./scripts/check_target_curl_usage.sh "$DIR"`, `./scripts/check_katana_usage.sh "$DIR"`, `./scripts/check_collection_health.sh "$DIR"`, and `./scripts/check_surface_coverage.sh "$DIR"` first; if any fail, stop and report the duplicate IDs, count mismatch, raw runtime bypasses, collection failures, or unresolved surfaces instead of normalizing silently

=== INTERIM REPORTS ===

Condensed: Progress Summary, Findings So Far, Next Steps. Skip narrative and appendix.

=== FINDING CONSOLIDATION ===

Before generating the report:
1. RENUMBER: Map all FINDING-XX-NNN to sequential FINDING-001 ~ FINDING-N, ordered by severity
   (CRITICAL first, then HIGH, MEDIUM, LOW, INFO).
2. DEDUP: If two agents reported the same vulnerability (same endpoint + same technique),
   merge into one finding. Credit both agents in "Discovered by".
3. SEVERITY AUDIT: If a finding is purely incremental info from an already-reported vuln
   (e.g., reading another file via same XXE), fold it into the parent finding as sub-evidence,
   do not count as separate finding.
4. REMOVE NON-FINDINGS: Drop entries that record working security controls or empty responses
   as "findings." These are observations, not vulnerabilities.
