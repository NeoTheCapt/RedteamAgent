## Phase 1: Bug Detection & Fix

Start by analyzing the latest fixed-target runs and their logs/artifacts before creating or deleting anything.

### Per-Target Rules

#### OKX (https://www.okx.com)

OKX runs are long-lived observation runs. They should stay `running` continuously — `completed` is NOT a normal state for OKX.
- If the OKX run is `queued` or `running`, leave it alone — this is normal.
- If the OKX run is `completed`, `failed`, `error`, `stopped`, `cancelled`, or `timeout`, the controller has already deleted it and created a fresh replacement. Investigate WHY the run terminated and fix the underlying bug in the RedTeam Agent code if needed.

#### Local / Juice Shop (http://127.0.0.1:8000)

Local runs are finite — they are expected to reach `completed` status.
- If the local run is still `queued` or `running`, leave it running. Do not score an incomplete run.
- If the local run is `completed`, proceed to Phase 2 for challenge score analysis.
- If the local run is `failed`, `error`, `stopped`, `cancelled`, or `timeout`, fix the underlying bug in the RedTeam Agent code, delete the broken run, and create one fresh replacement.

### Bug detection rules

- If `Observed path types` is too sparse, or most observed paths are not crawler-derived (`katana` / `katana-xhr`), treat that as a crawler-bug investigation trigger and dig deeper instead of skipping.

### Code change discipline (CRITICAL — past violations caused recall to regress from 18 to 1)

The following rules exist because prior Phase 2 cycles added cumulative complexity that collapsed recall. Every code change MUST follow these constraints:

- **Simplify, don't complexify.** If your fix adds more lines than it removes, justify why. Prefer removing unnecessary conditions over adding new ones.
- **One change per gap.** Do not stack multiple conditional branches for one problem. Pick the single most impactful fix.
- **Never bloat agent prompts.** Agent prompts (source-analyzer, vulnerability-analyst, operator) are read by an AI that gets confused by walls of conditional text. Keep prompts concise and directive. If a prompt paragraph exceeds 3 sentences, it is too long.
- **Fix bugs in scripts, not in prompts.** If the agent is doing something wrong, fix the script logic or the tool — do not add a 5-sentence "do not do X, unless Y, except when Z" paragraph to the prompt. Prompt patches are the #1 cause of regression.
- **Measure net line delta.** Before committing, run `git diff --stat`. If agent prompt/script files grew by more than 20 net lines, reconsider — you are likely over-engineering.
- **Do not add new tools/scripts for marginal gains.** A new 200-line helper script that solves one edge case is worse than a 2-line fix to an existing script.
- **Revert if unsure.** If you cannot clearly explain how your change improves recall for a specific unsolved challenge, do not make it.

### Score analysis rules

- The challenge score is ground truth (from Juice Shop API). Do not question it.
- Focus on improving recall of low-difficulty challenges first (difficulty 1-3).
- Higher-difficulty challenges (4-6) often require business logic or multi-step attacks that may not be feasible for automated scanning — don't force them.
- Compare current recall against `## Challenge Score History`. If recall is regressing, treat that as high-priority.
- Do NOT commit yet after Phase 2 — Phase 3 still needs to run.

7. If you cannot clearly explain "this change will help solve [specific challenge] because [mechanism]", do not make the change.

### Strictly prohibited changes

The following types of changes are **absolutely prohibited** because they inflate the Juice Shop score without improving the agent's real capability against other targets:

- **Hardcoded endpoints/paths** — Do not add specific URLs like `/rest/user/login`, `/ftp/`, `/api/Challenges`, `/redirect`, `/b2b/v2/orders` to agent prompts, skills, or scripts. The agent must discover endpoints through reconnaissance, not from a pre-programmed list.
- **Hardcoded payloads** — Do not embed specific attack strings (e.g., `' OR 1=1--`, `<iframe src="javascript:alert(1)">`, specific product IDs or user IDs) that only work against Juice Shop.
- **Challenge-aware detection** — Do not add rules that recognize Juice Shop challenge patterns (e.g., "if response contains 'challenge solved'", "look for score-board endpoints", "check /api/Challenges for progress").
- **Target fingerprint shortcuts** — Do not add "if Express framework detected, try X" or "if Juice Shop detected, do Y" branching logic.
- **Known-answer injection** — Do not encode knowledge of Juice Shop solutions (e.g., "the admin password is admin123", "use null byte in file upload", "the coupon code is...") into any agent file.
- **Disguised specificity** — Do not write rules that appear generic but are crafted to match only Juice Shop (e.g., "always check for files at /ftp/" or "always try XXE on file upload endpoints that accept XML"). A truly general rule should be derivable from OWASP methodology without knowledge of the specific target.

**The litmus test:** If you removed the Juice Shop instance and pointed the agent at a different web application (e.g., DVWA, WebGoat, HackTheBox), would your changes still improve the agent's performance? If not, the change is target-specific and must not be made.
